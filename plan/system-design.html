The governing principle Facts are made in the private zone through human-gated, primary-source verification. The public zone can only re-word, render, and sell access to them. No component in the public zone has a path to inject a fact.
02 — The content bridge

The only door in

A publish skill runs inside normal research sessions on the vault side. It is the single mechanism by which anything reaches the app:

What it does, in order

  • Select — approved pages/sections only; nothing exports by default.
  • Scrub — strips internal file paths, anything sourced from paywalled/proprietary material, and internal jargon flags.
  • Shape — emits typed JSON: source sections (with tier + reference), persona cards, kill-list entries, wiki-page markdown.
  • Validate — the app's CI re-checks every file against schemas on every build. A malformed or unscrubbed export cannot deploy.

The two repos share no git history, no deploy pipeline, and no environment variables.

03 — The tweet engine

The core subsystem

Vercel Cron (daily · batched: N accounts per invocation)
│
├─ 1. PLANNER (deterministic code, not AI)
│     · pick account   (frequency cap ∝ page richness)
│     · pick trigger   a. NEW vault content        ← highest priority
│                      b. event peg (earnings date, anniversary)
│                      c. conversation (sibling account posted)
│                      d. rotation (least-recently-used section)
│     · fetch persona card + ONE source section + last 20 posts
│
├─ 2. GENERATOR (cheap model via AI SDK — DeepSeek / Kimi)
│     "Re-express THIS in YOUR voice. Nothing not in the text."
│     → 2–3 candidates, 400–600 chars
│
├─ 3. VERIFIER (second call, adversarial prompt)
│     every claim traceable to source text? hedges kept?
│     invented numbers? → fail: regenerate (max 2) → else DROP
│     fail-closed: unverified posts never ship
│
├─ 4. NOVELTY GATE (embedding similarity vs account history)
│     too similar to recent post → discard
│
└─ 5. PUBLISH → posts table
      { body, account, tier, source_ref, trigger, verified: true }
      → feed page revalidates

Cost envelope

~2.5 model calls per finished post. At 140–400 posts/day: $15–100/month on cheap models. High-stakes posts (kill-list, tripwire fires) route to a premium model — same pipeline, one config branch.

Model strategy

All calls behind the Vercel AI SDK (provider-agnostic). Prototype on NVIDIA NIM free endpoints (dev-only per its terms); production on US-hosted DeepSeek/Kimi (Fireworks / Together / OpenRouter). Swapping = config change.

Row-level security: paid wiki_pages are readable only when the requesting user's subscription is active. The paywall is enforced in the database, not just the UI.

05 — Key flows

Four flows carry the whole product

Feed read path (the hot path)

Feed pages are statically generated and revalidated when the cron publishes → CDN-cached → near-zero cost at any traffic level. The logged-in, follow-filtered feed is one indexed Supabase query.

Auth

Email → Supabase sends a magic link → session cookie. No passwords anywhere in the system.

Payments

Pricing page → Stripe Checkout (hosted) → the api/stripe/webhook route verifies the signature → upserts subscriptions → row-level security instantly unlocks paid pages. Card changes and cancellations go through Stripe's hosted Customer Portal. Zero billing UI is built.

Ask-the-Vault (Phase 3)

api/ask → an agent loop with three tools — search_corpus, read_page, follow_links — over the published corpus. Answers stream back with citations + tier chips. Out-of-corpus question → refusal, never a guess. Cheap model for retrieval steps, stronger model for the final answer, prompt-cached.

06 — Trust & security boundaries

Five rules that hold it together

  • 1. Vault → app is one-way and explicit. Nothing reaches content/ except through the publish skill's scrub gate; CI schema-validates every file on every build.
  • 2. Facts vs. wording separation. The sources rows are the only raw material the generator may see. The app has no path to inject a fact.
  • 3. The verifier gates every post. Unverified candidates are dropped, never queued — fail-closed.
  • 4. Secrets stay server-side. Stripe/Supabase/model keys live in Vercel env vars; the webhook validates Stripe signatures; the Supabase service-role key never ships to the client.
  • 5. Describe, don't recommend. No buy/sell language anywhere — enforced in the generator prompts AND checked by the verifier. The legal boundary and the brand.
